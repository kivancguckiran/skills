# Contract-First Monorepo

Use this skill for TypeScript monorepos where shared contracts define API boundaries before backend and clients are updated.

Do not use it for single-package apps, one-off UI-only edits, or repositories where the API contract intentionally lives only in the backend.

## Orientation

1. Read the nearest `AGENTS.md`.
2. Confirm package manager from workspace files, lockfiles, and `package.json#packageManager`.
3. Identify the package boundary:
   - shared contracts/schemas/types
   - backend/API/worker/server routes
   - database schema/migrations
   - web client
   - mobile client
4. Inspect root scripts and package scripts before choosing validation commands.

## Inputs

Before editing, identify:

- the API or schema shape being changed
- packages that import or generate from that shape
- persistence/auth impact, if any
- the narrowest verification command available in the repo

## Change Order

For endpoint or API shape changes:

1. Update shared route constants, Zod schemas, request/response types, and exported contract types first.
2. Update backend route or server function next.
3. Update DB schema and migration when persistence changes.
4. Update web and mobile API wrappers.
5. Update UI screens or hooks last.
6. Add or update tests at the layer where behavior changed.

For database changes:

1. Update the source schema model.
2. Add a migration using the repo's existing migration naming and tooling.
3. Keep auth, ownership, RLS, permission checks, and audit logging aligned with local rules.
4. Run migration/status/scenario tests when available.

## Guardrails

- Do not duplicate schemas across packages; keep one source of truth.
- Do not expose service-role keys or privileged clients to browser/mobile code.
- Keep generated files such as route trees untouched unless regenerated by the repo tool.
- Prefer feature-local extraction before repo-wide abstraction.
- Preserve existing route registration style; do not introduce a framework router into a hand-rolled Worker without explicit need.
- Use explicit web/mobile wrappers when that is the repo pattern, even if they look similar.
- If contracts are generated, update the source and run the generator instead of editing generated output by hand.
- If multiple package managers appear, follow the lockfile and local instructions closest to the repo root.

## Verification

Choose the narrowest meaningful command:

- shared contract/types only: package build or typecheck
- backend route/auth/queue/cron: backend tests plus typecheck
- database/migration/RLS: migration status/scenario/data tests
- web UI: web build, typecheck, or UI tests
- mobile UI: mobile typecheck/start/build command from the repo
- broad cross-package change: repo verify/test-all script when present

If env vars, Docker, Supabase, Cloudflare, or network access are missing, state the skipped check and run the next local check that still proves the change.

## Output

When finished, report:

- contract/schema files changed
- backend, database, and client layers updated
- verification command run, or the exact missing prerequisite
